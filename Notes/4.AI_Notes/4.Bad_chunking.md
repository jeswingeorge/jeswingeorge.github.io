# Chunking

## Bad Chunking
**Bad chunking** occurs when a document is split into text segments (chunks) arbitrarily—such as by strict character or token counts—without respecting semantic, structural, or logical boundaries. This fractures cohesive information across chunk boundaries, delivering incomplete context to the Large Language Model (LLM) and causing synthesis failures during the generation phase.

---

### Analysis Steps

To diagnose and fix bad chunking, an AI engineer must break down document structure systematically:

1. **Structural Analysis**: Inspect document types (PDFs, Markdown, JSON) to identify logical boundaries like headers, paragraphs, lists, or tables.
2. **Semantic Cohesion Evaluation**: Check if individual sentences or paragraphs that rely on each other for meaning are being separated.
3. **Retrieval vs. Synthesis Profiling**: Evaluate if retrieval fails because a query hits a partial chunk, or if generation fails because the LLM receives truncated context.

---

### Code & Outputs

### The Failure Mode: Naive Character Chunking

Below is an example of a naive chunker splitting a critical piece of operational data right down the middle because it strictly enforces a fixed character limit.

```python
# Simulated Document Context
document = (
    "Company Policy Update 2026: The standard probation period for all new engineering hires "
    "is exactly 3 months. However, for Senior AI Engineers, the probation period is extended to 6 months."
)

# Naive Chunker: Split strictly every 100 characters without overlap
chunk_size = 100
chunk_1 = document[:chunk_size]
chunk_2 = document[chunk_size:]

print("--- BAD CHUNKING OUTPUT ---")
print(f"Chunk 1: '{chunk_1}'")
print(f"Chunk 2: '{chunk_2}'")

```

**Why this fails in RAG:**
If a user asks, *"What is the probation period for a Senior AI Engineer?"*, a vector search will likely retrieve **Chunk 2** because it contains the exact keywords. However, Chunk 2 contains fragmented text: `", the probation period is extended to 6 months."`

Without the subject context established in Chunk 1, the LLM has no idea *whose* probation period is 6 months, leading to a hallucination or an empty answer.

---

### The Solution: Semantic & Parent-Child Chunking

To mitigate this, we move away from naive slicing and adopt **Recursive Character Text Chunking** combined with a **Parent-Child (Small-to-Large) Retrieval** strategy using `langchain` and `uv` package-compatible syntax.

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

# 1. Mitigation Strategy: Recursive Chunking with Overlap
# This respects structural boundaries (\n\n, \n, " ") and maintains context via overlap.
recursive_splitter = RecursiveCharacterTextSplitter(
    chunk_size=120,
    chunk_overlap=30,  # Retains context from the previous chunk
    length_function=len,
    separators=["\n\n", "\n", ".", " ", ""]
)

smart_chunks = recursive_splitter.split_text(document)

print("\n--- MITIGATED CHUNKING OUTPUT (Recursive with Overlap) ---")
for i, chunk in enumerate(smart_chunks):
    print(f"Smart Chunk {i+1}: '{chunk}'")

```

### Visuals

When structural information is highly layout-dependent (like tables or financial reports), naive chunking completely breaks down. Below is a structural mapping of how advanced pipelines handle this via **Parent-Child Chunking**:

```
[ Raw Parent Document: Full Section on Hiring Policies ]
                         │
         ┌───────────────┴───────────────┐
         ▼                               ▼
[ Small Child Chunk 1 ]       [ Small Child Chunk 2 ]
(Sentences 1-2: Vector Index) (Sentences 3-4: Vector Index)
         │
         ▼ (User Query matches Child Chunk 2)
[ Pipeline Action: Map Child ID back to Parent ID ]
         │
         ▼
[ Context Fed to LLM: Full Parent Document Section ]

```

*Instead of feeding the fractured Child Chunk directly to the LLM, the pipeline uses the highly optimized child chunk for vector matching, but swaps it out to pass the broader, structurally intact Parent Context to the LLM generation step.*

---

### Recommendations

1. **Switch to Recursive Splitting**: Replace standard `CharacterTextSplitter` with `RecursiveCharacterTextSplitter` which uses a fallback list of separators to keep paragraphs and sentences together.
2. **Implement Parent-Child (Small-to-Large) Retrieval**: Index small chunks (e.g., 100-200 tokens) for highly accurate semantic vector search, but link them in your database to larger parent blocks (e.g., 500-1000 tokens) that get sent to the LLM.
3. **Use Markdown/HTML Parsers**: If your source data contains headers, use structural splitters like `MarkdownHeaderTextSplitter` so that metadata (e.g., `{"Header_1": "Probation Policies"}`) is appended to every single sub-chunk.

---

### Validation

* **Primary Metric**: **Context Recall** and **Faithfulness** via automated evaluation frameworks.
* **Ablation Test**: Benchmark your RAG system's answer accuracy on edge-case questions where the answer relies on data spanning across paragraph transitions. A healthy chunking strategy will show a near-flat performance curve regardless of where a sentence boundary falls.

## `chunk_size` and `chunk_overlap`

There is no single "golden number" for chunking configurations; rather, `chunk_size` and `chunk_overlap` must be mathematically and semantically tuned based on the **LLM's context window limits**, the **embedding model's token capacity**, and the **structural density** of your target data. Over-tokenizing leads to context overload, while under-tokenizing causes information fragmentation (bad chunking).

---

### Analysis Steps

To engineer optimal chunk configurations, a systematic RAG pipeline follows these steps:

1. **Determine Bottlenecks**: Identify the strict token limit of your embedding model (e.g., `text-embedding-3-small` caps at 8191 tokens, but optimizes semantically around 256–512 tokens).
2. **Analyze Information Density**: Evaluate data types. High-density text (legal contracts, financial statements) requires smaller chunk sizes with larger overlaps. Low-density prose (novels, generic documentation) allows for larger chunks.
3. **Establish Baseline Heuristics**:
* **`chunk_size`**: Typically ranges from **256 to 1024 tokens**.
* **`chunk_overlap`**: Typically set to **10% to 20%** of the chunk size to ensure sentences split across boundaries retain context.



---

### Code & Outputs

Here is a comprehensive script using `langchain-text-splitters` configured with a production-grade, parameter-driven recursive splitter. We use `tiktoken` to ensure chunk calculations are based on *tokens*, not characters (a critical error in naive pipelines).

```python
import tiktoken
from langchain_text_splitters import RecursiveCharacterTextSplitter

def tune_rag_chunks(text: str, target_chunk_size_tokens: int = 512, overlap_percentage: float = 0.2):
    """
    Simulates token-aware chunking configurations matching Cookiecutter Data Science standards.
    """
    # 1. Initialize tokenizer to measure actual token boundaries
    tokenizer = tiktoken.get_encoding("cl100k_base")
    
    # Calculate overlap size dynamically
    target_overlap_tokens = int(target_chunk_size_tokens * overlap_percentage)
    
    # 2. Heuristic conversion: 1 token ~ 4 characters for English text base
    # Text splitters use character length fundamentally, so we scale our parameters safely
    char_size = target_chunk_size_tokens * 4
    char_overlap = target_overlap_tokens * 4
    
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=char_size,
        chunk_overlap=char_overlap,
        length_function=len,
        separators=["\n\n", "\n", ".", " ", ""]
    )
    
    chunks = splitter.split_text(text)
    
    print(f"--- Pipeline Chunk Metrics [Target: {target_chunk_size_tokens} tokens] ---")
    for idx, chunk in enumerate(chunks[:3]):  # Sample first 3 chunks
        token_len = len(tokenizer.encode(chunk))
        print(f"Chunk {idx+1}: {token_len} tokens | Character length: {len(chunk)}")
        
    return chunks

# Sample raw payload text
sample_corpus = (
    "In deep learning architecture, transformer blocks utilize multi-head self-attention mechanisms. " * 50
)
_ = tune_rag_chunks(sample_corpus)

```

| Strategy | Chunk Size | Chunk Overlap | Best Used For | Risk Profile |
| --- | --- | --- | --- | --- |
| **Micro-Chunking** | 128–256 tokens | 25–50 tokens | Key-value data, structural logs, granular FAQs. | Information fragmentation, loss of holistic narrative. |
| **Standard/Balanced** | **512 tokens** | **50–100 tokens** | **General corporate docs, Wikis, academic papers.** | Sweet spot for modern embeddings (`bge`, `OpenAI`). |
| **Macro-Chunking** | 1024–2048 tokens | 100–200 tokens | Broad synthesis, summarization tasks, continuous narratives. | Induces **Context Overload**, dilutes embedding vector vectors. |

---

### Recommendations

#### 1. Match Configuration to Your Embedding Model

If your embedding model was trained on short paragraphs (like older sentence-transformers), keeping chunks at **256 tokens** maintains semantic accuracy. If using modern models (`Vertex AI text-embedding`, `text-embedding-3`), a **512-token** baseline yields exceptional semantic density.

#### 2. Implement the 10%–20% Overlap Rule

Never set overlap to 0. A sliding window of 10% to 20% (e.g., a 512-token chunk gets a ~50-100 token overlap) guarantees that nouns, acronyms, and conditional clauses (e.g., *"However, this only applies if..."*) are captured completely by at least one vector.

#### 3. Move Beyond Static Configurations

If your documents are structurally complex, stop relying purely on token sizes. Implement **Parent-Child Chunking**:

* Store **Child Chunks** (128 tokens) in your Vector Store for high-precision retrieval matching.
* Link them to **Parent Chunks** (1024 tokens) stored in a Document Store to pass a broad context to the LLM.

---

## Validation

To validate if your chunk size is correct, perform a **RAG Triad Ablation Test**:

1. Run evaluation pipelines (using frameworks like `Ragas`) across three configurations: `[256/32]`, `[512/64]`, and `[1024/128]`.
2. Measure **Context Recall** (did retrieval catch the answer?) and **Context Precision** (how much junk text did you force the LLM to read?).
3. Select the smallest configuration that maintains a Context Recall score of $>0.90$.
